# AI 에이전트 이력서 업데이트 가이드 (2025 FE)

이 문서는 `Resume with AI` 프로젝트에서 이력서 내용을 자동 갱신할 때 품질을 보장하기 위한 기준 문서다.  
목표는 "그럴듯한 문장"이 아니라, 채용 관점에서 통과 가능한 **증거 기반 이력서**를 만드는 것이다.

## 1) 좋은 이력서의 정의

좋은 이력서는 아래 3가지를 동시에 만족한다.

1. **빠르게 이해된다**: 6~10초 스캔에서 핵심 강점이 보인다.
2. **기술적 깊이가 보인다**: 무엇을 했는지보다 왜/어떻게 했는지가 드러난다.
3. **비즈니스 임팩트가 보인다**: 결과가 수치 또는 검증 가능한 변화로 표현된다.

## 2) 2025 채용 스크리닝 전제

1. **모바일/ATS 친화성**: 싱글 컬럼, 텍스트 기반 PDF 기준으로 읽힌다.
2. **키워드 나열만으로는 부족**: 기술 키워드와 성과가 같은 문장에 있어야 한다.
3. **AI 냄새 감점**: 추상 형용사/번역투 문장/근거 없는 미사여구는 신뢰를 떨어뜨린다.

## 3) 섹션별 작성 기준

### 3.1 헤더/타이틀

1. 이름 + 직무 + 차별화 포지셔닝을 한 줄로 명확히 쓴다.
2. "신입 프론트엔드 개발자"처럼 일반적인 표현만 쓰지 않는다.
3. 성능/안정성/데이터 기반 개선 같은 가치 키워드를 포함한다.

예시:

`비즈니스 임팩트를 우선하는 3년 차 프론트엔드 엔지니어`

### 3.2 요약(Summary, 3~5줄)

아래 3요소를 순서대로 포함한다.

1. 연차 + 핵심 직무
2. 기술적 강점(구체 명사 중심)
3. 협업 방식/엔지니어링 태도

금지:

1. "열정적인", "성실한" 같은 추상 형용사
2. 근거 없는 "최고", "혁신", "완벽"

### 3.3 기술 스택(Skills)

규칙:

1. 도구를 **범주화**해서 쓴다.
2. 숙련도를 점수/막대그래프로 표현하지 않는다.
3. 최신 트렌드 키워드는 "나열"이 아니라 사용 맥락과 연결한다.

권장 카테고리:

1. Languages: JavaScript (ES6+), TypeScript
2. Frameworks: React, Next.js (버전 명시)
3. State: Zustand/Jotai + TanStack Query
4. Testing: Vitest/Jest, Playwright/Cypress, Storybook
5. Build/DevOps: Vite/Webpack, GitHub Actions, Docker

### 3.4 경력/프로젝트(핵심)

모든 불렛은 아래 구조를 따른다.

`Problem -> Action(기술 선택 이유 포함) -> Result(수치/검증 가능한 변화)`

좋은 예:

`레거시 렌더링 병목으로 LCP 3.1s가 발생하던 화면에 Code Splitting과 이미지 최적화를 적용해 LCP 1.4s로 개선, 전환율 12% 상승`

나쁜 예:

`성능 최적화를 진행했고 사용자 경험을 크게 개선함`

기여도 표기 규칙:

1. `%` 대신 **책임 범위**를 명시한다.
2. 예: `결제 플로우 프론트엔드 설계/구현 전담`, `디자인 시스템 토큰 구조 설계 주도`

### 3.5 포트폴리오/GitHub

README 최소 요건:

1. 해결한 문제 한 줄 요약
2. 데모 링크
3. 기술 선정 이유
4. 실행 방법
5. 트러블슈팅 로그

코드 품질 신호:

1. 컨벤셔널 커밋
2. 명확한 폴더 구조
3. 테스트 코드(유닛/E2E)
4. 하드코딩 최소화, 의미 있는 변수명

## 4) 2025 FE 핵심 키워드 (우선 반영)

1. Next.js App Router, Server Components, Server Actions, Streaming SSR
2. TypeScript 고급 타입, 런타임 검증(Zod)
3. 서버 상태 관리(TanStack Query)
4. Web Vitals(LCP/CLS) 기반 성능 개선
5. 모노레포, 디자인 시스템, CI/CD 자동화

## 5) AI 작성 가이드 (자동 업데이트용)

원칙:

1. 초안 생성보다 **사실 정리/교정**에 AI를 쓴다.
2. 모든 문장은 근거 데이터(실제 작업 로그, PR, 수치)와 연결한다.
3. 결과 수치가 없으면 "관찰 가능한 변화"를 명시한다.
4. **절대 허위 수치/허위 역할/허위 경험을 추가하지 않는다.**
5. 근거 없는 내용은 생성하지 말고, 확인 불가 항목은 비워두거나 "측정/검증 필요"로 표기한다.

프롬프트 규칙:

1. 입력은 반드시 사실 목록으로 시작한다.
2. 출력 형식은 `Problem -> Action -> Result` 불렛으로 강제한다.
3. 금지어 목록을 함께 준다.

권장 금지어:

`혁신적인`, `열정적인`, `다양한 경험`, `최고의`, `완벽한`

휴머니징 체크:

1. 번역투 접속사 과다 사용 제거
2. 문장 길이 리듬 다양화
3. 실제 팀 맥락/의사결정 이유 1문장 이상 포함

## 6) 품질 게이트 체크리스트

업데이트 전/후 아래 항목을 모두 점검한다.

### 필수(MUST)

- [ ] PDF/모바일에서 읽기 쉬운 싱글 컬럼 구조인가
- [ ] 상단 3~5줄 요약에 차별화 포인트가 있는가
- [ ] 각 경력 불렛이 `Problem -> Action -> Result`를 포함하는가
- [ ] 결과가 수치 또는 검증 가능한 변화로 표현되었는가
- [ ] 모든 수치/성과/역할에 근거(로그, PR, 문서, 측정값)가 있는가
- [ ] 존재하지 않는 프로젝트/역할/기술을 추가하지 않았는가
- [ ] 기술 키워드가 성과 문장 안에서 맥락적으로 쓰였는가
- [ ] 기여도를 `%` 대신 역할/책임 범위로 표현했는가
- [ ] 추상 형용사/미사여구를 제거했는가
- [ ] AI 번역투 표현을 사람 문장으로 다듬었는가
- [ ] 오탈자/문법 오류가 없는가
- [ ] README에 문제/데모/선정이유/실행/트러블슈팅이 있는가
- [ ] 클론 코딩 중심이 아닌 실제 문제 해결 사례가 있는가
- [ ] 최신 FE 트렌드 키워드가 실제 경험과 연결되어 있는가

### 권장(SHOULD)

- [ ] 핵심 성과 3개가 이력서 상단 절반에 배치되어 있는가
- [ ] 성능 지표(LCP/CLS/번들 크기) 중 1개 이상을 포함했는가
- [ ] 협업 신호(코드리뷰, 문서화, 멘토링)를 포함했는가
- [ ] 실패/제약과 의사결정 근거를 1개 이상 담았는가
- [ ] 회사별 맞춤 키워드(예: 기본기, 임팩트, 협업문화)를 반영했는가

## 7) 자동 업데이트 워크플로우 권장 순서

1. **사실 수집**: 작업 로그, PR, 성능 측정값, 배포 지표를 수집한다.
2. **초안 생성**: 각 경험을 `Problem -> Action -> Result`로 1차 작성한다.
3. **AI 교정**: 간결화/중복 제거/문장 정돈만 수행한다.
4. **품질 게이트**: MUST 체크리스트 100% 충족 여부를 확인한다.
5. **최종 검수**: 사람이 읽고 "내가 실제로 한 일처럼 들리는지" 확인한다.

## 8) 안티패턴 모음

1. 기술 스택만 길게 나열하고 실제 성과가 없는 이력서
2. "성능 개선", "협업 강화"처럼 결과 없는 추상 표현
3. AI가 쓴 티가 나는 상투 문장 반복
4. 클론 코딩 프로젝트를 핵심 포트폴리오로 제시
5. 팀 프로젝트에서 본인 기여 범위가 불명확한 서술

## 9) 빠른 템플릿

경력 불렛 템플릿:

`[문제 상황] 때문에 [지표/비즈니스 영향]이 발생 -> [기술 선택 + 이유]로 해결 -> [정량/정성 결과] 달성`

요약 템플릿:

`[연차/직무]로서 [핵심 기술/도메인]에 강점이 있으며, [대표 성과]를 만들었다. [협업/품질 관점의 일하는 방식]을 통해 지속 가능한 제품 개발에 기여한다.`

## 10) 2026 리팩터링 표준

2026-02-15 리팩터링부터 아래 기준을 기본 규칙으로 적용한다.

1. **결과 우선 문장**: 모든 불릿은 결과/영향으로 시작하고, 수행한 행동은 뒤에 배치한다.
2. **내부 용어 배제**: 내부 클래스명·파이프라인 명칭·해상도 같은 맥락 약한 세부치는 일반 표현으로 치환한다.
3. **불릿 압축**: 불릿당 핵심 메시지는 1개로 제한하고, 20~25단어 이내를 목표로 중복 문장을 병합한다.
4. **PAR 요약 유지**: 프로젝트 본문은 `Problem -> Action -> Result` 구조를 유지하되 각 항목은 1~2줄로 요약한다.
5. **핵심 영문 병기**: 제목과 핵심 성과 불릿(수치/임팩트 중심)에만 짧은 영어 병기를 추가한다.
6. **수상/활동 연결**: 학습/수상 항목은 "무엇을 배웠는지"보다 "실무 프로젝트에 어떻게 적용했는지"를 한 줄로 연결한다.
7. **자격증 표기 규칙**: 자격증 항목에는 점수·등급 등 검증 가능한 수준 정보를 함께 표기한다.
8. **사실성 우선**: 새 수치/성과를 추정 생성하지 않으며, 검증 불가 정보는 삭제하거나 방향성 표현으로 축약한다.
