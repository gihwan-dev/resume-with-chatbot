PRD: 포트폴리오 상세 페이지 스토리텔링 UI/UX 개편

1. 개요 및 목표

배경: 기존 포트폴리오 상세 페이지의 'Problem -> Action -> Result' 수평 분할(가로썰기) 구조로 인해, 특정 문제와 그에 따른 해결책 간의 인과관계(스토리)가 단절되는 현상 발생.

목표: 'Hook First' 구조와 'Story Thread' 기반의 수직적(세로썰기) 내러티브 UI를 도입하여, 지원자의 기술적 사고 흐름과 문제 해결 능력을 면접관이 직관적으로 이해할 수 있도록 개선.

주요 타겟: 포트폴리오를 평균 몇 십 초 이내로 빠르게 스캐닝하는 채용 담당자 및 시니어 엔지니어.

2. 핵심 UI/UX 전략 및 패턴 (리서치 기반)

A. Hook First ("영화 트레일러" 구조)

상세 페이지 진입 시 가장 먼저 노출되는 Hero 섹션에 최종 비즈니스/성능 임팩트를 배치합니다.

이 영역에만 시선을 집중시키기 위한 **'강조형 카드 UI'**를 허용하여, 면접관이 "어떻게 이걸 해냈지?"라는 호기심을 갖게 한 뒤 본문(Context)으로 유도합니다.

B. Story Thread (이슈 기반 문제-해결 페어링)

Problem, Action, Result를 흩뿌리지 않고, **[특정 기술적 이슈 1개 = 1개의 스레드]**로 묶어 화면을 구성합니다.

(UX 원칙) 글의 흐름이 끊기지 않도록 본문 영역에서 닫힌 형태의 테두리나 박스(Card) UI는 전면 배제합니다. 대신 좌측의 얇은 타임라인 라인(Connecting Line)과 넉넉한 여백을 활용해 하나의 독립된 서사가 물 흐르듯 읽히게 만듭니다.

C. 전환 문장 (Transition Sentences) 도입

단순 불릿 나열을 지양하고, Problem에서 Action으로 넘어가는 사이에 **"왜 이런 아키텍처적 결정을 내렸는지"**를 설명하는 전환 문장(옅은 배경색의 인용구 형태)을 추가하여 사고 과정을 드러냅니다.

D. Before & After 내러티브 블록

Action 파트 내부에 'Before & After' 비교 UI를 배치하여, 변화의 크기를 시각적(아키텍처 다이어그램, 코드 스니펫 등)으로 증명합니다.

3. 화면 구조 설계 (Top-to-Bottom)

3.1. Hero Section (도입부)

페이지 타이틀: 프로젝트 명칭 (예: 고객 특화 DB 모니터링 대시보드 아키텍처 재설계)

한 줄 요약: 프로젝트의 핵심 가치를 설명하는 문장

Impact Badges: 핵심 성과 수치를 강조하는 2~3개의 카드 (수치, 레이블, 짧은 설명 포함)

3.2. Context Section (배경)

해당 프로젝트에서 직면했던 당시 상황, 팀 규모, 제약 사항 등을 줄글 형태로 서술하여 본격적인 스토리 시작 전 배경 지식을 제공합니다.

3.3. Story Threads (이슈 해결 과정 - 반복 영역)

프로젝트 내에서 해결한 2~3개의 핵심 이슈를 테두리 없이 타임라인과 여백만으로 구분하여 렌더링합니다.

이슈 타이틀: 직면한 가장 큰 문제 요약 (예: 파편화된 폴링과 렌더링 경합 문제)

Problem (현상): 붉은색 테마를 활용해 당시 발생했던 에러 상황이나 성능 저하 현상을 불릿 형태로 명시

Thought Process (사고 과정): 옅은 보라색/푸른색 인용구 스타일을 활용해 문제를 해결하기 위해 어떤 고민을 했는지 서술 (전환 문장)

Action (해결): 푸른색 테마를 활용해 기술적 도입 내용과 아키텍처 개선 사항을 명시

Interactive Element (선택): Before/After 토글 뷰어를 삽입하여 시각적 증거 제시

Result (결과): 녹색 테마를 활용해 해당 액션으로 인해 얻은 즉각적인 효과 서술

3.4. Retrospective Section (회고)

프로젝트를 마치며 개발자로서 얻은 인사이트나 아쉬운 점, 팀의 표준으로 자리 잡은 내용 등을 서술하며 마무리합니다.

4. 데이터 모델 정의 (콘텐츠 스키마 요구사항)

이 구조를 렌더링하기 위해 프로젝트 데이터는 다음과 같은 논리적 구조를 가져야 합니다.

기본 정보: title, summary

핵심 성과 (Impacts): value(수치), label(제목), description(설명)

스토리 스레드 (Threads - 다중 배열 구조):

issueTitle: 이슈 명칭

problems: 발생했던 문제점 목록 (배열)

thoughtProcess: 문제 해결을 위한 사고의 흐름 (문자열)

actions: 구체적 해결 방안 목록 (배열)

result: 해당 이슈를 해결하여 얻은 개별 성과 (문자열)

회고 (Retrospective): lessonsLearned

5. 프로젝트 개발 마일스톤

실행 라벨: [SEQUENTIAL](순차), [PARALLEL:PG-n](같은 그룹끼리 병렬)
분해 기준: 컨텍스트 윈도우 친화 단위(한 번에 구현/검증 가능한 크기)

Phase 1. [SEQUENTIAL] Story Thread 데이터 계약 확정

[ ] 스토리텔링 상세 페이지 스키마 정의

목표: Hook First, Context, Story Threads, Retrospective를 안정적으로 렌더링할 수 있는 데이터 계약을 확정합니다.

검증:

impacts, threads, thoughtProcess, lessonsLearned의 필수/옵션 규칙이 정의됩니다.

기존 Problem/Decision/Result 본문을 새 구조로 매핑하는 규칙이 합의됩니다.

스키마 검증 실패 시 오류 기준(누락/타입 불일치/빈 배열)이 명확해집니다.

[ ] 마이그레이션 스파이크(대표 케이스 1건)

목표: 기존 케이스를 새 스키마로 전환할 때의 난이도와 누락 리스크를 사전에 제거합니다.

검증:

대표 케이스 1건이 신규 구조로 시범 변환됩니다.

변환 체크리스트(수치 근거, 전환 문장, 이슈-액션 페어링)가 작성됩니다.

변환 불가 항목의 보완 원칙(요약/삭제/추가 작성)이 정리됩니다.

Phase 2. [PARALLEL:PG-1] 콘텐츠 마이그레이션 배치 A

[ ] 핵심 케이스 2건 Story Thread 리라이팅

목표: 스캔 친화성과 인과관계가 드러나는 서사 구조로 핵심 프로젝트 2건을 전환합니다.

검증:

각 케이스에 Impact 2~3개가 수치 기반으로 채워집니다.

각 케이스에 이슈 스레드 2~3개(Problem→Thought→Action→Result)가 완성됩니다.

Retrospective 문단이 배운 점 중심으로 정리됩니다.

Phase 3. [PARALLEL:PG-1] 콘텐츠 마이그레이션 배치 B

[ ] 잔여 케이스 Story Thread 리라이팅 및 품질 정합화

목표: 나머지 프로젝트를 동일한 톤/깊이/형식으로 마이그레이션해 전체 일관성을 확보합니다.

검증:

잔여 케이스가 동일 스키마로 전환됩니다.

용어/문장 톤/성과 지표 단위 표기가 통일됩니다.

누락된 사고 과정(전환 문장) 없는 상태가 됩니다.

Phase 4. [PARALLEL:PG-1] 핵심 UI 컴포넌트 구축

[ ] Impact Badge 카드 컴포넌트 구현 (상단 전용)

목표: Hero 영역에서 핵심 성과를 즉시 인지할 수 있는 강조형 카드 UI를 구현합니다.

검증:

카드가 수치, 레이블, 설명 3요소를 모두 렌더링합니다.

2~3개 카드 배치 시 데스크톱/모바일 레이아웃이 안정적입니다.

시각적 강조가 본문보다 우선 노출되도록 계층이 설계됩니다.

[ ] Story Thread 끊김 없는 타임라인 컴포넌트 구현 (본문 전용)

목표: 박스(Card) 형태를 배제하고, 여백과 타임라인 라인만으로 Problem-Thought-Action-Result를 물 흐르듯 읽히게 만듭니다.

검증:

닫힌 테두리(박스) 없이 좌측 연결 라인만으로 서사의 연속성이 표현됩니다.

Problem/Action/Result 상태 색상 구분이 일관됩니다.

Thought Process 인용구의 옅은 배경색이 시각적 환기(Transition) 역할을 수행합니다.

Phase 5. [SEQUENTIAL] 상세 페이지 통합 및 내비게이션 정합성

[ ] Top-to-Bottom 내러티브 페이지 조립

목표: Hero → Context → Story Threads → Retrospective 순서로 상세 페이지를 재구성합니다.

검증: 첫 화면에서 임팩트가 확인되며, Context 이후 스레드가 자연스럽게 이어집니다.

[ ] 목차/딥링크/섹션 활성화 동작 갱신

목표: 새 섹션 구조에서도 TOC, 해시 이동, active state가 정확히 동작하도록 맞춥니다.

검증: 해시 링크 진입 시 올바른 섹션이 표시되며 스크롤 스파이가 일치합니다.

Phase 6. [PARALLEL:PG-2] 인터랙티브 증거 블록 고도화

[ ] Before & After 토글을 Story Thread에 표준 삽입

목표: 글만으로 어려운 구조 변화는 비교 UI로 증명할 수 있게 만듭니다.

검증: Action 컨텍스트 안에서 자연스럽게 표시되며 콘텐츠 길이 차이에도 레이아웃이 깨지지 않습니다.

Phase 7. [PARALLEL:PG-2] 반응형 및 인쇄 최적화

[ ] 모바일/데스크톱 반응형 완성

목표: 빠른 스캔 사용자를 위해 화면 크기별 정보 우선순위와 가독성을 최적화합니다.

검증: 모바일에서 Hero→Thread 순서와 간격이 읽기 친화적으로 유지됩니다.

[ ] @media print 인쇄 품질 개선

목표: PDF 출력 시 타임라인 라인과 배경 표현이 깔끔하게 보이도록 최적화합니다.

검증: 인쇄물에서 불필요한 장식 요소가 숨겨지고 텍스트 중심의 가독성이 유지됩니다.

Phase 8. [SEQUENTIAL] 품질 검증 및 완료 기준 충족

[ ] 회귀 테스트 및 접근성 점검

목표: 개편 후 기존 핵심 사용자 흐름과 접근성 품질을 유지합니다.

검증: 포트폴리오 딥링크/목차/프린트 관련 테스트가 통과합니다.

[ ] PRD 수용 기준(AC) 최종 확인

목표: 문서 요구사항과 구현 결과를 일치시켜 릴리즈 판단 가능 상태로 만듭니다.

검증: 팀 리뷰에서 “수십 초 스캔” 시나리오 기준으로 흐름 파악이 가능하다는 합의를 얻습니다.
