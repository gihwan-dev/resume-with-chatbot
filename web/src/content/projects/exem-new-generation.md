## TL;DR

차세대 DB 모니터링 제품에서 대시보드 위젯 추가 구조를 정리했습니다.
전역 스토어가 불필요한 화면은 Context API 기반 지역 상태로 전환했습니다.
대시보드 레이아웃 저장을 브라우저 저장소에서 DB 영구 저장 구조로 하루 만에 옮겼습니다.

## 왜 이 문제를 먼저 풀어야 했나

이 프로젝트에서 먼저 풀어야 했던 문제는 기능 부족이 아니라 제품 복잡성 증가였습니다.

- 위젯이 늘어날수록 위젯 빌더 수정 범위가 커졌습니다.
- 전역 스토어가 필요 없는 화면까지 전역 스토어가 남발되면서, 초기화와 초기값 주입 코드가 반복됐습니다.
- 대시보드 레이아웃 정보를 브라우저 저장소에 저장하고 있었는데, DB에 영구 저장해야 하는 요구가 생겼고 이를 1~2일 안에 처리해야 했습니다.
- SQL 에디터, 실행계획 비교, 메타데이터 조회 요소가 페이지별로 따로 구현돼 재사용 기준이 없었습니다.

## 내가 해결한 복잡성은 네 가지였다

- 확장 복잡성: 신규 위젯이 늘어날수록 위젯 빌더 수정 범위가 커지는 문제
- 상태 생명주기 복잡성: 전역 스토어가 필요 없는 화면까지 전역 스토어가 남발되던 문제
- 저장소 마이그레이션 복잡성: 브라우저 저장소 기반 레이아웃 저장을 짧은 일정 안에 DB 영구 저장 구조로 전환해야 했던 문제
- 분석 상호작용 복잡성: SQL 분석 페이지에서 공통 컴포넌트 추출 기준이 없어 화면마다 구현이 갈리던 문제

## 주제 1. 위젯 추가 비용을 줄이기 위한 구조 정리

### 문제

위젯 빌더는 여러 인스턴스의 지표를 line/bar/area/scatter/table 시각화 방식으로 구성하는 폼입니다. 새 위젯 유형을 추가할 때마다 설정 폼, 검증, 미리보기 로직을 함께 수정해야 해 회귀 위험이 커졌습니다.

### 해결과 결과

위젯별 설정 등록부와 검증 스키마, 미리보기 연결 계층을 분리했습니다. line·area·bar·scatter·table 위젯을 추가할 때 위젯별 로직만 수정해도 되도록 바꿔 수정 범위를 줄였습니다.

## 주제 2. 상태 생성과 초기화를 한 패턴으로 맞추기

### 문제

전역 스토어가 필요 없는 화면까지 전역 스토어가 남아 상태 범위가 불필요하게 넓어졌고, 생성·초기화·초기값 주입 코드가 화면마다 반복됐습니다.

### 해결과 결과

불필요한 전역 스토어 20개를 Context API 기반 지역 상태로 전환했습니다. 스토어 생성 함수를 도입해 생성·초기화 절차를 공통화하고 초기값 주입 기준을 맞췄습니다.

## 주제 3. 대시보드 레이아웃 저장 방식을 영구 저장 구조로 바꾸기

### 문제

대시보드 레이아웃 정보를 브라우저 저장소에 저장하고 있었는데, DB에 영구 저장해야 하는 요구가 생겼고 이를 1~2일 안에 처리해야 했습니다.

### 해결과 결과

브라우저 저장소와 사내 영구 저장소를 모두 키-값 저장 형태로 다룰 수 있다는 점에 맞춰 저장 API 인터페이스를 먼저 통일했습니다.
기존 저장 로직은 유지한 채 저장 대상만 교체해 하루 만에 마이그레이션을 마쳤습니다.

## 주제 4. SQL 분석 화면 공통 컴포넌트와 핵심 상호작용 구현

### 문제

SQL 에디터, 실행계획 비교, 메타데이터 조회 요소가 페이지별로 따로 구현돼 공통 컴포넌트 재사용 기준이 없었습니다.

### 해결과 결과

공통으로 쓰는 SQL 에디터, 실행계획 비교 UI, 메타데이터 탐색 컴포넌트를 식별해 추출·추상화했습니다. 각 페이지에서는 필요한 컴포넌트를 조합해 쉽게 재사용할 수 있게 정리했습니다.
- 실행계획 비교 UI: Best/Worst 자동 비교, drag-and-drop 전환, 좌우 스크롤 동기화, diff highlighting, drill-down
- SQL 원문 뷰어: 포맷팅, 복사, 새 탭/Full Sheet, 바인드 변수 하이라이팅과 값 치환
- 메타데이터 탐색: table/index/partition 병렬 조회, 가변 컬럼, 파생 태그 기반 UI

실행계획 비교·원문 확인·메타데이터 탐색에서 필요한 핵심 동작을 컴포넌트 기준으로 정리해 페이지가 달라도 재사용할 수 있게 했습니다.

## 그래서 전체적으로 무엇이 달라졌나

기능을 늘렸다기보다 변경이 반복되는 지점을 더 예측하기 쉬운 방향으로 정리한 데 의미가 있습니다.

- 신규 위젯 추가 시 위젯별 설정·검증·미리보기 로직만 수정하도록 정리했습니다.
- 전역 스토어가 필요 없는 화면의 불필요한 전역 스토어 20개를 지역 상태로 전환했습니다.
- 대시보드 레이아웃 저장을 브라우저 저장소에서 DB 영구 저장 구조로 전환하되 기존 저장 인터페이스를 유지해 단기 마이그레이션 요구에 대응했습니다.
- SQL 분석 페이지에서 공통으로 쓰는 컴포넌트를 추출해 페이지별 재사용 기준을 맞췄습니다.

기능이 늘어날 때 어떤 지점을 수정해야 하는지 더 예측하기 쉬운 방향으로 정리했습니다.

## 어떻게 확인했나

- 위젯 빌더, 상태 생성 패턴, 영구 저장 전환, SQL 분석 핵심 상호작용을 시나리오 기준으로 확인했습니다.
- 신규 위젯 추가 시 위젯별 설정·검증·미리보기 로직 중심으로 수정 지점이 제한되는지 확인했습니다.
- 전역 스토어 20개를 지역 상태로 전환한 뒤 초기화와 초기값 주입 동작이 유지되는지 확인했습니다.
- 브라우저 저장소 기반 시나리오와 동일한 조건에서 저장 엔진 교체 이후에도 레이아웃 저장/복원 동작이 유지되는 것을 확인했습니다.
- SQL 에디터, 실행계획 비교, 메타데이터 탐색 컴포넌트가 페이지별 조합에서도 의도한 상호작용으로 동작하는지 확인했습니다.

## What I Learned

B2B 데이터 제품에서는 화면 수보다 화면 안 상호작용 정확도가 더 중요했습니다. 상태 관리도 전역/로컬 선택보다 언제 생성되고 어디까지 영향을 주는지부터 정리해야 읽히는 구조가 됩니다.
