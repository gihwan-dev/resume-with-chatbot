## TL;DR

DB 성능 모니터링 제품에서 신규 차트를 쉽게 추가할 수 있는 Widget Builder 구조를 설계했습니다. Zustand `storeFactory`로 상세 진입, 새 탭 진입, 영속 상태의 생성과 초기화 방식을 통일했습니다. Oracle SQL 분석 영역에서는 실행계획 비교, SQL 원문 보기, 메타데이터 탐색 기능을 구현했습니다.

## 문제 정의

새 차트를 추가할 때마다 공통 Builder 수정 범위가 커졌고, 상세 진입과 새 탭 진입마다 상태 초기화 방식이 달라 버그 위험이 있었습니다. Oracle SQL 분석 영역에서는 실행계획 비교, SQL 원문 보기, 메타데이터 탐색 화면마다 필요한 상호작용이 달라 구현 복잡도가 높았습니다.

## 핵심 의사결정

차트별 설정과 검증 로직은 공통 Builder에서 분리했고, 상태는 화면 생명주기에 맞춰 생성하도록 바꿨습니다. SQL 분석 영역은 실행계획 비교, SQL 원문 보기, 메타데이터 탐색처럼 구현 범위가 분명한 화면 단위로 나눠 만들었습니다.

## 구현 전략

차트별 설정·검증·미리보기 로직은 공통 Builder 코드에서 분리했습니다. 상태는 스토어 생성 시 초기값을 넣는 방식으로 통일했고, SQL 분석 화면에서는 실행계획 비교 UI, SQL 원문 뷰어, 메타데이터 탐색 화면을 각각 구현했습니다.

## 검증 및 결과

신규 차트 추가 시 공통 Builder 로직 수정 범위를 줄였고, 상세 진입과 새 탭 진입에서 상태 생성 방식이 일관돼 초기화 관련 버그 위험을 낮췄습니다. SQL 분석 화면에서는 복잡한 비교 UI, 텍스트 뷰어, 동적 컬럼 메타데이터 화면을 안정적으로 구현할 수 있는 기반을 만들었습니다.

## What I Learned

B2B 데이터 화면에서는 화면 개수를 늘리는 것보다, 각 화면에서 필요한 상호작용을 정확히 구현하는 것이 더 중요했습니다. 상태와 공통 로직도 이름보다 언제 생성되고 어디까지 영향을 주는지부터 정리해야 읽기 쉬운 구조가 됩니다.
