## TL;DR

DB 성능 모니터링 제품에서 신규 차트를 쉽게 추가할 수 있는 Widget Builder 구조를 설계했습니다. Zustand `storeFactory`로 상세 진입, 새 탭 진입, 영속 상태의 생성과 초기화 방식을 통일했습니다. Oracle SQL 분석 영역에서는 실행계획 비교, SQL 원문 보기, 메타데이터 탐색 기능을 구현했습니다.

## 문제 정의

새 차트를 추가할 때마다 공통 Builder 수정 범위가 커졌고, 상세 진입과 새 탭 진입마다 상태 초기화 방식이 달라 버그 위험이 있었습니다. Oracle SQL 분석 영역은 실행계획 비교, SQL 원문 보기, 메타데이터 탐색 화면마다 필요한 상호작용이 달라 구현 복잡도가 높았습니다.

- 새 차트를 추가할 때마다 공통 폼과 검증 로직 수정 범위가 커졌습니다.
- 상세 진입과 새 탭 진입에서 상태 생성·초기값 설정 방식이 달라 수동 초기화 코드가 반복됐습니다.
- SQL 분석 화면은 실행계획 비교, SQL 원문 보기, 메타데이터 탐색마다 상호작용 패턴이 달라 화면별 구현 복잡도가 높았습니다.

## 핵심 의사결정

1. 공통 Builder를 계속 수정하는 방식 대신 `visualization` 기준 registry + Zod 판별 유니온 + preview adapter로 차트별 설정·검증·미리보기 로직을 분리했습니다.
2. 전역 상태 하나를 공유하는 방식 대신 화면 진입 시 `storeFactory`로 생성/초기값 주입 패턴을 통일했습니다.
3. SQL 분석 영역은 단일 복합 화면이 아니라 실행계획 비교, SQL 원문 보기, 메타데이터 탐색으로 구현 단위를 분리했습니다.

## 구현 전략

### 구조 설계

- line·area·bar·scatter·table 차트 추가 시 공통 Builder 수정 범위를 최소화했습니다.
- `storeFactory`를 도입해 Provider-local/global/persist를 하나의 생성 패턴으로 통합했습니다.
- 스토어 생성 시 초기값 주입 방식을 고정해 20개 스토어의 상태 생성 기준을 일치시켰습니다.

### SQL 분석 화면 구현

- 계층형 실행계획 비교 UI를 구현해 Best/Worst 자동 비교, drag-and-drop 전환, 좌우 스크롤 동기화, diff highlighting, 세부 조건 drill-down을 지원했습니다.
- Monaco 기반 SQL 원문 뷰어를 구현해 SQL 포맷팅, 복사, 새 탭/Full Sheet 열기, 바인드 변수 하이라이팅과 값 치환을 지원했습니다.
- 메타데이터 탐색 화면을 구현해 table/index/partition 정보를 병렬 조회하고, 가변 컬럼과 파생 태그 기반 UI로 구조화했습니다.

## 검증 및 결과

- Widget Builder, 스토어 생성 패턴, SQL 분석 화면의 주요 상호작용을 Story/VRT 시나리오와 구현 코드 기준으로 검증했습니다.
- 신규 차트 추가: 공통 Builder 로직 수정 범위 최소화
- 상태 관리: 화면 진입 방식이 달라도 생성·초기화 방식 일관화
- SQL 분석 화면: 실행계획 비교 + SQL 원문 보기 + 메타데이터 탐색 구현
- 신규 차트 추가 시 공통 Builder 전체를 다시 건드리는 일을 줄였고, 초기화 관련 버그 위험을 낮췄습니다.

## What I Learned

B2B 데이터 화면에서는 화면 개수를 늘리는 것보다, 각 화면에서 필요한 상호작용을 정확히 구현하는 것이 더 중요했습니다. 상태와 공통 로직도 이름보다 언제 생성되고 어디까지 영향을 주는지부터 정리해야 읽기 쉬운 구조가 됩니다.
