## TL;DR

차세대 DB 모니터링 제품의 복잡성을 확장 구조, 상태 생명주기, 분석 UX 세 축으로 정리해 기능이 늘어나도 유지 가능한 기반을 만들었습니다.

## 왜 이 문제를 먼저 풀어야 했나

이 프로젝트에서 먼저 풀어야 했던 문제는 기능 부족이 아니라 제품 복잡성 증가였습니다.

- 차트가 늘어날수록 공통 Builder 수정 범위가 커졌습니다.
- 상세 진입과 새 탭 진입마다 상태 생성·초기화 방식이 달라 수동 코드가 반복됐습니다.
- SQL 분석 화면은 실행계획 비교, SQL 원문 확인, 메타데이터 탐색이 서로 다른 상호작용을 요구했습니다.

## 내가 해결한 복잡성은 세 가지였다

- 확장 복잡성: 신규 차트가 늘어날수록 공통 Builder 수정 범위가 커지는 문제
- 상태 생명주기 복잡성: 진입 방식마다 상태 생성과 초기화 흐름이 달라지는 문제
- 분석 상호작용 복잡성: SQL 분석 기능이 분절돼 탐색 흐름이 끊기는 문제

## 주제 1. 차트 추가 비용을 줄이기 위한 구조 정리

### 무엇이 문제였나

새 차트를 추가할 때마다 공통 폼과 검증 로직을 함께 수정해야 해 회귀 위험이 커졌습니다.

### 어떤 판단을 했나

공통 Builder를 계속 확장하는 대신, 차트별 설정·검증·미리보기 로직을 Builder 밖으로 분리하기로 했습니다.

### 어떻게 구현했나

`visualization` 기준 registry + Zod 판별 유니온 + preview adapter 구조를 두고 차트별 설정·검증·미리보기 로직을 분리했습니다.

### 무엇이 달라졌나

line·area·bar·scatter·table 추가 시 공통 Builder 전체를 다시 건드리는 부담을 줄였습니다.

## 주제 2. 상태 생성과 초기화를 한 패턴으로 맞추기

### 무엇이 문제였나

상세 진입과 새 탭 진입에서 상태 생성·초기값 설정 방식이 달라 수동 초기화 코드가 반복됐습니다.

### 어떤 판단을 했나

전역 상태 하나로 밀어붙이지 않고, 화면 생명주기에 맞게 store를 생성하되 생성 패턴은 통일하기로 했습니다.

### 어떻게 구현했나

`storeFactory`를 도입해 Provider-local/global/persist를 하나의 생성 패턴으로 통합하고, 스토어 생성 시 초기값 주입 방식을 고정해 20개 스토어 기준을 맞췄습니다.

### 무엇이 달라졌나

진입 방식이 달라도 상태 생성과 초기화 흐름을 같은 방식으로 읽을 수 있게 됐고 초기화 관련 버그 위험을 낮췄습니다.

## 주제 3. 복잡한 SQL 분석을 실제로 탐색 가능한 UX로 만들기

### 무엇이 문제였나

SQL 분석 기능을 단순히 화면별로 나누면 사용자가 비교·확인·탐색을 오가며 흐름이 끊겼습니다.

### 어떤 판단을 했나

기능 나열이 아니라 분석 흐름 중심으로 실행계획 비교, SQL 원문 확인, 메타데이터 탐색을 연결하기로 했습니다.

### 어떻게 구현했나

- 실행계획 비교 UI: Best/Worst 자동 비교, drag-and-drop 전환, 좌우 스크롤 동기화, diff highlighting, drill-down
- SQL 원문 뷰어: 포맷팅, 복사, 새 탭/Full Sheet, 바인드 변수 하이라이팅과 값 치환
- 메타데이터 탐색: table/index/partition 병렬 조회, 가변 컬럼, 파생 태그 기반 UI

### 무엇이 달라졌나

복잡한 SQL 분석 작업을 분절된 기능 집합이 아니라 실제 탐색 가능한 흐름으로 제공할 기반을 만들었습니다.

## 그래서 전체적으로 무엇이 달라졌나

결과는 기능 3개를 추가한 것이 아니라 제품 복잡성을 관리 가능한 구조로 바꾼 데 있습니다.

- 차트 확장 비용을 줄였습니다.
- 상태 생성과 초기화 규칙을 통일했습니다.
- SQL 분석 UX를 탐색 흐름 중심으로 정리했습니다.

기능이 늘어나도 구조가 무너지지 않는 방향으로 기반을 재정리했습니다.

## 어떻게 확인했나

- Widget Builder, store 생성 패턴, SQL 분석 핵심 상호작용을 Story/VRT 시나리오와 구현 코드 기준으로 검증했습니다.
- 신규 차트 추가 시 공통 Builder 로직 수정 범위 축소 여부를 확인했습니다.
- 상태 생성·초기화 흐름 일관성과 SQL 분석 핵심 탐색 동작을 시나리오 단위로 확인했습니다.

## What I Learned

B2B 데이터 제품에서는 화면 수보다 화면 안 상호작용 정확도가 더 중요했습니다. 상태 관리도 전역/로컬 선택보다 언제 생성되고 어디까지 영향을 주는지부터 정리해야 읽히는 구조가 됩니다.
