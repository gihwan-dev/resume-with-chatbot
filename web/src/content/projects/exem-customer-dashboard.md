#### 모니터링 대시보드 아키텍처 개편 및 React 마이그레이션 (Monitoring Dashboard Re-architecture & React Migration)

**Problem**

* 카드형 UI 한계로 300개 이상 이기종 DB 상태를 한 화면에서 파악하기 어려워, 초기 장애 인지와 우선순위 판단이 지연됐습니다.
* 전역 의존과 고객사별 분기 누적으로 변경 영향 범위가 커져, API 전환 시 유지보수 리스크가 빠르게 증가했습니다.

**Action**

* 단일 화면 데이터 밀도를 높인 그리드 중심 UI와 동적 레이아웃 구성을 도입해, 고객사별 관제 우선순위를 같은 제품 구조에서 대응했습니다.
* 레거시 의존을 분리한 React 프로젝트로 이관하고 핵심 기능 동등성 검증을 선행해, 마이그레이션 과정의 기능 회귀 위험을 줄였습니다.

**Result**

* 장애 인지 시간이 약 10초에서 3초로 70% 단축되어 대응 속도가 개선됐습니다 (Reduced alert recognition time by 70%).
* 레거시 API 의존을 제거하고 테스트 기반 이관을 완료해, 고객사 요구 변화 대응과 릴리즈 안정성을 함께 확보했습니다.

#### 실시간 데이터 폴링 아키텍처 표준화 및 네트워크 최적화 (Polling Standardization & Network Optimization)

**Problem**

* 페이지 전환 뒤에도 호출이 남아 백그라운드 트래픽이 누적됐고, 필터 즉시 반영 로직이 분산되어 데이터 정합성 관리가 어려웠습니다.
* API별 폴링 주기와 재실행 규칙이 화면마다 달라, 동일한 운영 시나리오에서도 네트워크 동작 예측이 어려웠습니다.

**Action**

* 중앙 폴링 매니저로 호출·구독·해제를 표준화해 컴포넌트별 타이머 관리 코드를 제거하고, 화면 생명주기에 맞춘 데이터 흐름으로 정리했습니다.
* 필터 변경 시 즉시 재실행 규칙을 공통 처리로 이동해, 갱신 타이밍 제어를 화면 코드가 아닌 공통 계층에서 관리했습니다.

**Result**

* 비동기 타이머 보일러플레이트가 줄어 유지보수 복잡도가 낮아졌고, 화면 전환과 필터 조작 시 네트워크 패턴이 일관되게 동작했습니다.
* 예외 요청과 지연 갱신 이슈를 줄여 데이터 정합성과 프론트엔드 안정성을 강화했습니다 (Improved consistency and frontend stability).
