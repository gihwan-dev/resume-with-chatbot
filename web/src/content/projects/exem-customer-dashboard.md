## TL;DR

300~3,000대 인스턴스를 다루는 운영 대시보드에서 카드형 2-depth 흐름을 고밀도 그리드 기반 1-depth 허브로 재설계해 한 화면 비교와 즉시 이동이 가능하도록 바꿨습니다. 화면마다 흩어진 폴링과 상태 저장 로직은 선언적 폴링 제어와 중앙 저장소 구조로 일원화했습니다. `useFrozenData`, `useDeferredValue`, memo 최적화로 폴링·리사이즈가 겹치는 구간의 렌더 총량을 150회에서 5회로 96.7% 줄이고, INP를 400ms대에서 100ms대로 안정화했습니다.

## 왜 이 문제를 먼저 풀어야 했나

운영자는 300개 이상 인스턴스를 한 화면에서 빠르게 비교하고 이상 징후를 판단해야 했지만, 정보 밀도·상태 구조·제품 연계 동선·인터랙션 안정성이 동시에 병목인 상태였습니다.

- Vue 2 레거시 화면마다 폴링 규칙이 흩어져 운영 재현성이 낮았습니다.
- 서로 영향을 주는 필터/조회 상태가 분산돼 수정 시 화면 간 파급 범위가 커졌습니다.
- 저장/복원/마이그레이션 로직이 반복돼 같은 변경을 여러 지점에서 수정해야 했습니다.
- 카드형 UI는 300개 이상 인스턴스를 한 화면에서 비교하기에 정보 밀도가 낮았습니다.
- 제품 연계 동선이 깊어 허브 화면에서 즉시 대응 흐름이 끊겼습니다.
- 폴링과 리사이즈가 겹치는 구간에서 입력 응답이 흔들렸습니다.

## 핵심 설계 판단

1. `restart/start/stop` 인터페이스의 PollingManager로 화면별 타이머를 묶어 폴링 제어를 선언적으로 통합했습니다.
2. 필터/조회 상태와 저장/복원 흐름을 중앙 저장소로 모으고 버전 기반 마이그레이션을 적용했습니다.
3. API 전면 교체 시점에 Vue 연장 대신 React + 사내 디자인 시스템 재구축을 선택했습니다.
4. 카드형 2-depth 구조를 고밀도 그리드 1-depth 허브로 바꿔 한 화면 비교와 즉시 이동 흐름을 확보했습니다.

## 어떻게 운영 판단 흐름을 다시 설계했나

### 갱신 규칙과 상태 구조

- PollingManager를 기준으로 `restart/start/stop` 흐름을 통일해 화면별 분산 폴링을 선언적으로 제어했습니다.
- 서로 영향을 주는 필터/조회 상태를 중앙 경로에서 다루고 저장/복원 흐름을 한 구조로 정리했습니다.
- 저장 상태에는 버전 기반 마이그레이션을 적용해 스키마 변경 시 호환 경로를 고정했습니다.

### 고밀도 허브 UX

- API 전면 교체 시점에 React와 사내 디자인 시스템으로 대시보드를 재구축했습니다.
- 카드형 2-depth 구조를 고밀도 그리드 기반 1-depth 허브로 전환해 한 화면 비교 밀도를 높였습니다.
- 위젯에서 RTM/PA를 즉시 실행할 수 있게 연결해 제품 간 연계 이동 단계를 줄였습니다.

### 인터랙션 안정성

- 드래그/리사이즈 상태를 전역 스토어로 추적하고 종료 후 200ms 지연 해제를 적용해 레이아웃 정착과 데이터 갱신 재개가 겹치지 않게 했습니다.
- `useFrozenData`는 드래그/리사이즈 중 마지막 안정 데이터 스냅샷을 유지해 폴링 응답이 조작 중 화면을 흔들지 않게 했습니다.
- `useDeferredValue`는 새 데이터 렌더 우선순위를 낮춰 사용자 입력 반응을 먼저 보장했습니다.
- `useFrozenLayout`과 주요 컴포넌트 `memo`를 적용해 레이아웃 흔들림과 불필요한 리렌더를 줄였습니다.

## 무엇이 달라졌나

- 드래그 중 렌더: **150회 -> 0회**
- 총 렌더: **150회 -> 5회**
- 총 렌더 기준: **96.7% 감소**
- INP: **400ms대 -> 100ms대 안정화**

## What I Learned

운영 대시보드는 폴링 정책, 영속 상태 구조, 정보 밀도, 이동 깊이를 함께 설계해야 효과가 유지됩니다. 실시간 제품에서도 인터랙션 구간은 최신 데이터 반영보다 사용자의 조작 안정성이 더 중요할 수 있습니다. 프로파일링으로 병목을 수치화하고 동일 시나리오 회귀 기준을 고정해야 개선 효과를 지속할 수 있습니다.
