#### 모니터링 대시보드 아키텍처 개편 및 React 마이그레이션

**Problem**

* 기존 카드형 UI의 낮은 데이터 밀도로 인해 **300개 이상의 이기종 DB(Oracle, Tibero 등)** 상태를 한눈에 파악하기 어려움 (Maxgauge 연계 시 2-Depth 진입 필요).
* **루트 전역 객체**에 대한 강한 의존성으로 인해 앱 전반의 결합도가 높아 유지보수가 어려운 상태.
* 기존 API가 제거되고 신규 API로 대체되는 상황에서, 3개 고객사의 상이한 요구사항을 하나의 대시보드 내 Feature Flag로 관리하며 코드 복잡도가 한계에 도달함.

**Action**

* 카드 UI를 리스트 기반 **데이터 그리드**로 전환. 가상 스크롤링을 적용하여 단일 화면 내 인스턴스 노출량을 4배(50개 → 200개) 확장하고, 제품 연계 Depth를 1단계로 축약.
* **전역 객체 결합도 해소**를 위해 React 기반 신규 프로젝트로 분리 결정. 기존 Vue2 환경의 핵심 기능(알림 시스템 등)에 **Playwright E2E 테스트**를 선행 구축하여 기능의 동일성을 보장하며 안정적으로 이관.
* **동적 레이아웃 구성 시스템**을 도입하여 고객사별 관제 우선순위에 따라 대시보드를 드래그 앤 드롭으로 재구성할 수 있도록 지원하고, 레이아웃 상태를 로컬 스토리지에 영속화.
* 복잡한 Feature Flag 구조를 프로젝트 분리 및 컴포넌트화를 통해 정리하여 잠재적인 사이드 이펙트 제거.


**Result**
* 장애 인지 시간 **70% 단축 (약 10초 → 3초)** 및 주요 기능 접근 경로 단순화.
* 레거시 API 의존성 완전 제거 및 테스트 자동화를 통한 마이그레이션 성공.
* 사내 디자인 시스템 표준 준수로 공통 컴포넌트 재사용률 향상.
* 고객사별 요구사항 대응 속도 개선 및 대규모 인스턴스 환경에서도 **60fps** 이상의 안정적인 렌더링 성능 유지.

#### 실시간 데이터 폴링 아키텍처 표준화 및 네트워크 최적화

**Problem**

* 페이지 전환 시 기존 호출이 정지되지 않아 불필요한 백그라운드 요청이 지속되는 현상 발생.
* 폴링 주기가 API별로 상이하고, 특정 필터 적용 시 즉시 데이터를 갱신(폴링 주기 재시작)해야 하는 트리거 로직이 각 컴포넌트에 파편화되어 유지보수성이 저하됨.

**Action**

* 전역 Polling Manager를 설계하여 단일 인터벌 내에서 API를 호출하고, 구독한 컴포넌트들에게 데이터를 브로드캐스트하는 **구독/발행(Pub-Sub) 구조**로 전환.
* 컴포넌트 마운트 시 구독, 언마운트 시 구독 해제를 자동화.
* 필터 변경 등 즉각적인 갱신이 필요한 시점에 매니저 내부 타이머를 초기화하고 즉시 재실행하는 기능을 구현하여 외부 컴포넌트의 제어 복잡도 해소.

**Result**

* 개별 컴포넌트에서 비동기 타이머를 관리하던 보일러플레이트 코드를 제거하고, 데이터 흐름을 선언적으로 개선하여 관련 코드 복잡도 대폭 감소.
* 페이지 전환 및 필터 조작 시 발생하는 예외적인 네트워크 패턴을 정돈하여 프론트엔드 안정성 및 데이터 정합성 강화.
