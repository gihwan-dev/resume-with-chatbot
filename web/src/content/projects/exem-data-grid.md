## TL;DR

table 기반 구조의 기능 조합 충돌을 해결하기 위해 통합 테스트로 동작을 고정한 뒤 div 렌더링으로 전환하고, 리사이즈 스펙 변경에 맞춰 Row absolute + Cell flex 구조로 재설계했습니다. Core - State - UI 분리와 호환표 기반 검증 체계를 정착시켜 DOM 노드 90% 감소, 리사이즈 처리 22ms에서 0.5ms 단축을 달성하면서 기능 조합 확장성을 함께 확보했습니다.

## 문제 정의

초기 table 기반 구조에서는 virtualization, 고정 컬럼, 그룹 헤더, 리사이즈를 함께 구현할 때 구조 충돌이 반복됐고, 대량 데이터 환경에서 렌더링 비용과 열 조작 비용이 동시에 증가했습니다. 공용 그리드 특성상 화면별 보정이 아니라 기반 구조 자체를 바꾸는 접근이 필요했습니다.

- table의 행/열 동기화 레이아웃이 virtualization, 고정 컬럼, 그룹 헤더의 독립 위치 제어와 충돌해 기능 조합마다 예외 처리가 필요했습니다.
- 대량 데이터에서 렌더링 비용과 열 조작 계산 비용이 함께 커져 화면 응답성이 흔들렸습니다.
- 기능 수가 늘수록 조합별 스펙 충돌과 회귀 범위를 수동으로 관리하기 어려웠습니다.
- 공용 컴포넌트라 화면별 임시 분기보다 공통 제약을 명시하는 구조 재설계가 우선 과제였습니다.

## 핵심 의사결정

1. table 유지 대신 div 기반 렌더링 구조로 전환했습니다.
2. Cell absolute 1차 구조에서 Row absolute + Cell flex 2차 구조로 재설계했습니다.
3. 임시 조건문 누적 대신 리사이즈/고정 컬럼/그룹 헤더 동작을 제약 조건 규칙으로 명시했습니다.
4. Core - State - UI 계층 분리로 테스트 가능성과 확장성을 확보했습니다.

## 구현 전략

### 핵심 제약 조건 규칙

- 테이블 내부 빈 공간 최소화를 기본 원칙으로 두고, 예외 시나리오는 별도 규칙과 테스트로 관리합니다.
- 기본적으로 컬럼 너비 변경 영향은 우측 컬럼으로 전달하며, 마지막 리프 헤더는 컨테이너 전체 너비 계산에 직접 영향을 줍니다.
- 우측 고정 컬럼 리사이즈는 반대 방향(좌측 컬럼)으로만 영향이 전달되도록 분리합니다.
- 리사이즈 핸들은 리프 헤더에만 부여하고, 첫/마지막 리프 헤더는 부모 계층 수를 반영한 높이의 핸들을 제공합니다.
- 부모 그룹 헤더 너비는 자식 리프 헤더 너비 합으로 계산하고, 컨테이너는 상/중/하 및 좌/중/우 분할로 고정/스크롤 조합을 다룹니다.

### 시간 순서 구현 흐름

- 리팩토링 전후 동작을 통합 테스트로 선행 정비해 기존 스펙을 고정했습니다.
- 초기 table 기반 구조를 div 렌더링으로 전환하고 1차로 Cell absolute positioning을 적용했습니다.
- 리사이즈 시 인접 컬럼 재분배 스펙 변경에 맞춰 2차로 Row absolute + Cell flex 구조를 도입했습니다.
- 제어/비제어 인터페이스를 일관화하고 Custom Feature + Core - State - UI 계층으로 분리했습니다.
- 호환표를 기준으로 Storybook 시나리오, 브라우저 통합 테스트, 회귀 타깃 문서를 함께 운영했습니다.

## 검증 및 결과

- 정량 검증은 React Profiler, Chrome DevTools Performance, Performance API로 동일 시나리오를 반복 측정했습니다.
- 정성 검증은 기능 호환표를 기준으로 Storybook 시나리오, 브라우저 기반 통합 테스트, 회귀 타깃 문서를 함께 운영했습니다.
- DOM 노드: 약 **90% 감소**
- 리사이즈 처리: **22ms -> 0.5ms**
- 기능 조합 검증: 호환표 기반 문서·테스트·회귀 타깃 운영
- 대규모 데이터 화면에서도 성능과 유지보수성을 함께 확보했습니다.

## What I Learned

대규모 그리드는 레이아웃 모델, 제약 조건 모델링, 검증 체계를 함께 설계해야 지속 가능합니다. 공용 UI 인프라는 기능이 늘어날수록 구현 기법보다 스펙 관리 방식이 더 중요해집니다. 테스트와 문서는 사후 산출물이 아니라 복잡한 조합을 다루는 설계 도구가 됩니다.
