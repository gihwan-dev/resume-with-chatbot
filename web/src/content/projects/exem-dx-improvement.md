## TL;DR

전역 통신 규칙, ExtJS 이벤트 문맥, SQL 실행 추적, 실행 환경 구성의 암묵 규칙을 명시화·도구화해 레거시 유지보수의 착수 비용을 낮췄습니다.

## 문제 정의

프론트에서 SQL을 직접 실행하는 구조와 전역 객체 의존, ExtJS 컨텍스트 제약이 겹쳐 이슈 재현과 원인 추적이 개인 숙련도에 의존했습니다.

## 핵심 의사결정

화면별 임시 수정보다 공통 규칙(JSDoc/Helper), 진단 도구, 스타터킷을 먼저 정리해 변경 위험과 추적 비용을 함께 줄이는 방향을 선택했습니다.

## 구현 전략

`WS`·`IMXWS` 문서화, `this` 바인딩 컨텍스트 고정, SQL bind 공용화, 스타터킷 중앙화, 팀 공용 Sencha 진단 도구 구축으로 유지보수 경로를 고정했습니다.

## 검증 및 결과

SQL 응답 확인과 원인 추적 진입이 수동 패치 중심 방식 대비 즉시 수준으로 빨라졌고, 버전별 환경 구성 착수도 수 시간 단위 병목을 줄이는 방향으로 개선됐습니다.

## What I Learned

레거시 FE 개선은 “무엇을 새로 만들었는가”보다 “암묵 규칙을 얼마나 명시적 구조로 바꿨는가”가 유지보수 품질을 결정합니다.
