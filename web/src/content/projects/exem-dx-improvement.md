## TL;DR

레거시 FE의 핵심 병목이던 진단 가능성 부족 문제를 암묵 규칙 명시화와 진단 도구 공용화로 정리해 원인 추적 착수 비용을 줄였습니다.

## 문제 정의

전역 객체 의존, ExtJS 이벤트 문맥, 프론트 SQL 직접 실행 구조가 겹치면서 이슈 재현과 원인 추적이 사람 숙련도에 크게 의존하던 상태였습니다. 문제의 핵심은 기능 부족이 아니라 진단 가능성 부족이었습니다.

- 전역 `WS`·`IMXWS` 통신 규칙이 명시되지 않아 화면별 추측 기반 수정이 반복됐습니다.
- ExtJS 이벤트 콜백에서 `function` 문맥으로 인해 `this` 컨텍스트가 변하며 상태 판별 로직이 오동작했습니다.
- 프론트 SQL 응답 확인을 위해 코드에 `console.log`를 삽입해야 해 재현 속도와 안전성이 모두 떨어졌습니다.
- 레포/리소스/실행 절차가 분산돼 버전별 개발 환경 셋업에 수 시간이 소요됐습니다.

## 핵심 의사결정

1. 화면별 임시 수정 누적보다 전역 통신·SQL 규칙의 명시화를 우선했습니다.
2. 수동 콘솔 패치 대신 팀 공용 Sencha 진단 도구를 도입해 추적 경로를 표준화했습니다.
3. 분산 셋업 문서 운영 대신 스타터킷 중앙화로 버전별 실행 절차를 고정했습니다.

## 구현 전략

### 암묵 규칙 명시화

- 전역 `WS`·`IMXWS` 통신 모듈에 JSDoc을 정리해 공통 호출 규칙과 해석 기준을 문서화했습니다.
- Oracle 옵션에 따라 달라지는 SQL bind 치환 규칙을 `Exem.mixin.SQLBindHelper`로 공용화해 화면별 예외 처리를 줄였습니다.
- ExtJS 이벤트 콜백에서 `function` 문맥으로 `this`가 유실되던 지점을 추적하고 arrow function으로 컨텍스트를 고정했습니다.

### 진단 도구 공용화

- 팀 공용 Sencha 진단 도구를 구축해 SQL 응답을 실시간으로 확인하고 컴포넌트를 IDE로 바로 추적할 수 있게 했습니다.

### 환경 진입 비용 축소

- 흩어진 레포·리소스·실행 절차를 스타터킷으로 중앙화해 버전별 환경 구성을 빠르게 시작할 수 있도록 정리했습니다.

## 검증 및 결과

- 레거시 이슈 재현/원인 추적 흐름에서 기존 수동 방식(`console.log` 삽입, UI 텍스트 검색)과 도구 적용 이후 방식을 비교해 체감 지표를 기록했습니다.
- SQL 응답 확인: 수동 `console.log` 삽입 중심 확인에서 도구 기반 실시간 확인으로 전환
- 레거시 환경 셋업: 수 시간 소요되던 초기 구성 부담 완화
- 원인 추적 경로: UI 텍스트 검색 중심 추적에서 컴포넌트 직접 추적 흐름으로 전환
- 이슈 해결 자체를 도구가 대신하진 않지만, 원인 추적과 수정 진입 시간이 줄어 초기 병목이 완화됐습니다.

## What I Learned

레거시 FE 유지보수의 핵심은 기능 추가보다 암묵 규칙을 명시화해 진단 가능성과 수정 가능성을 확보하는 데 있습니다. 도구는 문제를 자동 해결하지 않지만, 사람 의존적 추적 과정을 재현 가능한 절차로 바꾸는 효과가 있습니다.
