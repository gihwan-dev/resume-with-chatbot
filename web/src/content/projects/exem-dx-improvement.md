#### Architecture Summary

- 다운로드 중심 검증 경로를 스트리밍 기반 허브로 전환했습니다.
- GitLab 내부에서 AI 보조 흐름이 닫히도록 협업 자동화를 설계했습니다.
- 디버깅 가시성과 환경 구성 절차를 도구화해 재현성을 높였습니다.
- 반복 병목을 표준 워크플로로 치환해 팀 생산성을 안정화했습니다.

#### Context

- 폐쇄망 환경에서 검증 결과 접근이 느려 피드백 루프가 지연됐습니다.
- 코드 리뷰 중 AI 보조 활용을 위해 외부 이동이 반복됐습니다.
- 레거시 환경 설정이 복잡해 온보딩과 재현 비용이 컸습니다.

#### Problem

- 검증 루프가 느려 릴리즈 의사결정 타이밍이 늦어졌습니다.
- 리뷰 문맥 전환으로 핵심 설계 검토 집중도가 떨어졌습니다.
- 디버깅과 온보딩이 개인 숙련도에 과도하게 의존했습니다.

#### Why It Was Hard

- 보안/망분리 제약으로 일반적인 SaaS 도입 패턴을 사용할 수 없었습니다.
- 리포트, 리뷰, 온보딩 병목이 서로 다른 시스템에 분산돼 있었습니다.

#### Architecture Decision

A안(기존 도구/절차 유지)은 단기 변경 비용이 낮지만 병목이 지속됐습니다. B안(내부 통합 자동화)은 구축 복잡도가 높았지만 검증·리뷰·온보딩 흐름을 동시에 개선할 수 있어 B안을 선택했습니다. 위험은 계약 표준화, 감사 로그, 단계적 전환으로 완화했습니다.

#### Implementation Strategy

1. 리포트 업로드/서빙 파이프라인을 설계해 다운로드 중심 확인 흐름을 제거했습니다.
2. GitLab Webhook + N8N 자동화를 정의해 리뷰 맥락 내 AI 협업을 표준화했습니다.
3. 디버깅 도구와 온보딩 CLI를 구축해 재현 가능한 운영 경로를 만들었습니다.
4. 접근 시간/문맥 전환/온보딩 완료 시간을 지표로 정의해 반복 측정했습니다.

#### Result

- 리포트 접근 시간: 3분 -> 5초 (30회 평균, p95 기준)
- 리뷰 중 외부 도구 이동: 0회
- 신규 온보딩 시간: 수 시간 -> 5분 (신규 인원 5회 반복 기준)
- 운영 영향: 검증 착수 속도와 리뷰 집중도가 개선돼 팀 피드백 루프가 빨라짐

#### What I Learned

- DX 개선은 기능 추가가 아니라 팀 반복 병목을 구조적으로 제거하는 설계 문제입니다.
- 제약이 큰 환경일수록 자동화 결과보다 운영 가능한 표준 절차 정의가 더 중요합니다.
